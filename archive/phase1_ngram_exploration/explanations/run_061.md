# Run 061 — all-past n=3 online component with slower 20000-token warmup

## Proposed change
Add `submissions/ngram_interpolated_n5_laplace03_recent_n3_w14_warm20000_win262144.py`. It keeps the current best Run 060 architecture and changes only the ramp schedule for the all-past online component:

```text
main model:          n=5, alpha=0.3, train+online counts
online component:    n=3, alpha=0.3, test-prefix-only counts
window:              262144 observed symbols (> official 200000-token prefix)
max mixture weight:  0.14
warmup:              20000 tokens instead of 10000
```

The final mixture weight is unchanged; only early trust in the test-prefix-only component is reduced.

## Source and evidence
- source: experiment observation — Runs 052-055 showed the online-only n=3 component improved as more past symbols were available, so early estimates may be the noisiest part of the component.
- source: experiment observation — Run 060 improved only barely at weight `0.14`, showing the steady-state weight is very flat and arguing against more narrow weight tuning as the next move.
- source: experiment observation — Run 058 showed excessive weight hurts, while Run 059 showed higher auxiliary order remains too noisy; a slower ramp targets early variance without changing the final low-order model.
- source: official/project rules — `AGENTS.md` and `COMPETITION_RULES.md` require strict sequential prediction and fix `N=200000`; a longer warmup uses only already observed symbols and does not change the official evaluator.
- source: course material — `lectures/5_04022026_Inequalities_study.md` ties excess log-loss to model mismatch; `lectures/13_10032026_UniversalCompressionI_LempelZiv_study.md` treats synchronized past history as evidence that strengthens with more observations; `lectures/15_17032026_UniversalCompressionIII_MinimaxRedundancy_study.md` motivates mixture hedges when source identity/evidence is uncertain.

## Course-material connection
This is a bias-variance adjustment over time. Early in the sequence, the test-prefix-only n=3 model has little synchronized history, so trusting it too quickly may add redundancy. A slower ramp keeps more probability on the train-heavy main model until the online component has stronger empirical evidence, while preserving the same final convex mixture.

## Hypothesis
If early all-past n=3 online estimates are noisy, a 20000-token warmup will lower average log-loss by delaying full auxiliary weight. If useful test-specific low-order evidence appears quickly, the slower warmup will underuse it and worsen versus Run 060 (`2.9719679315`).

## Risks
Only the first 20000 tokens are affected, so the effect may be small and public-practice-specific. Runtime and memory should be unchanged. The predictor remains self-contained, sequential, and does not use future symbols, external calls, or evaluator changes.
