## Source and evidence
- source: experiment observation — Run 064 auxiliary `alpha=0.2`, Run 065 auxiliary `alpha=0.1`, and Run 066 auxiliary `alpha=0.05` all improved, but the gain from `0.1` to `0.05` was smaller.
- source: experiment observation — Runs 052-055 showed the all-past low-order component becomes stable with full-prefix history, supporting sharper smoothing than the sparse main model.
- source: experiment observation — Run 033 showed sharp smoothing is unsafe for the full `n=5` main model, so only the dense auxiliary model is changed.
- source: official/project rules — `AGENTS.md` and `COMPETITION_RULES.md` require online prediction and fix `N=200000`; all probabilities use only past counts and preserve evaluator semantics.
- source: course material — `lectures/5_04022026_Inequalities_study.md` ties excess bits to cross-entropy/KL mismatch; `lectures/13_10032026_UniversalCompressionI_LempelZiv_study.md` treats synchronized past history as empirical evidence; `lectures/15_17032026_UniversalCompressionIII_MinimaxRedundancy_study.md` motivates robust mixtures that hedge empirical estimates.

## Course-material connection
This run pushes the empirical-history side of the bias-variance tradeoff. A very small pseudocount lets dense online contexts dominate, potentially reducing model mismatch, but it can add redundancy through rare-symbol surprise penalties. The main model mixture remains the safety hedge, so the predictor stays a valid sequential code.

## Hypothesis
If auxiliary `alpha=0.05` is still over-smoothed, `alpha=0.02` will improve over Run 066 (`2.9718226807`). If the optimum is near `0.05`, `0.02` will over-sharpen and worsen.

## Risks
This is an aggressive low pseudocount. Rare or unseen continuations in online contexts may receive too little probability, especially early despite the warmup. Runtime and memory should be unchanged. Any small gain should be validated because this is public-practice calibration.
