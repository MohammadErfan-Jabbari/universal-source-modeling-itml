# Run 071 — all-past n=3 alpha 0.01 weight 0.16 with warmup 30000

## Proposed change
Add `submissions/ngram_interpolated_n5_laplace03_recent_n3_recenta001_w16_warm30000_win262144.py`. It keeps the current best Run 069 final mixture weight and auxiliary smoothing, but slows the ramp to that weight:

```text
main model:          n=5, alpha=0.3, train+online counts
online component:    n=3, alpha=0.01, test-prefix-only counts
window:              262144 observed symbols (> official 200000-token prefix)
max mixture weight:  0.16

```

Only the auxiliary weight schedule changes; after warmup the model is the same convex mixture mass as Run 069.

## Source and evidence
- source: experiment observation — Run 069 showed the sharpened `alpha=0.01` auxiliary model benefits from more final mixture weight (`0.16`) than the old `0.14` setting.
- source: experiment observation — Run 070 showed `0.18` over-weights that auxiliary model, so the final weight should stay at `0.16` unless the schedule changes.
- source: experiment observation — Runs 061-063 found the old auxiliary setup preferred a finite warmup around `20000` tokens, and Run 062 showed `40000` was too slow. With sharper smoothing and higher final weight, an intermediate `30000` ramp may reduce early sparse-context penalties without underusing the auxiliary model too much.
- source: official/project rules — `AGENTS.md` and `COMPETITION_RULES.md` require strictly sequential online predictions over the official `N=200000` prefix; this schedule uses only the count of past observed test symbols.
- source: course material — `lectures/5_04022026_Inequalities_study.md` ties excess bits to cross-entropy/KL mismatch; `lectures/13_10032026_UniversalCompressionI_LempelZiv_study.md` treats synchronized past history as empirical evidence; `lectures/15_17032026_UniversalCompressionIII_MinimaxRedundancy_study.md` motivates mixtures that hedge uncertain source estimates.

## Course-material connection
Early in the sequence, the test-prefix-only low-order model has little synchronized history, so its sharp `alpha=0.01` estimates can be mismatched. A slower ramp is a universal-coding hedge: rely on the train-heavy long-context model while online evidence is weak, then increase mass on the empirical online source as history accumulates.

## Hypothesis
If the sharp, weight-`0.16` auxiliary model is too noisy during the first `20000` symbols, a `30000` warmup will improve over Run 069 (`2.9717929495`). If useful online evidence is already reliable by `20000`, the slower ramp will underuse it and worsen.

## Risks
This affects only the first 15% of the evaluated prefix, so any gain may be small. Earlier `40000` warmup was too slow under older settings, so `30000` may also underuse valid online evidence. Runtime and memory should be unchanged.
