# Run 010 — interpolated n=4 with weaker high-order confidence (C=32)

## Proposed change
Add `submissions/ngram_interpolated_n4_c32.py`, the same recursive interpolated n-gram family as Run 009 but with interpolation constant `C=32` instead of `16`. Larger `C` means each local higher-order estimate needs more observations before it dominates the lower-order mixture.

## Source and evidence
- source: experiment observation — Run 009 improved public-practice score to `2.9817296585`, the first successful direction, using count-dependent interpolation over orders 0..3 with `C=16`.
- source: experiment observation — Runs 002-008 showed hard/thresholded high-order selection and simple smoothing/adaptation toggles were worse, so continue tuning mixtures rather than returning to hard backoff.
- source: ideas backlog — `autoresearch.ideas.md` proposes count-dependent mixture weights and exponential confidence `weight_order_k ∝ count_k / (count_k + c)`.
- source: course material — `lectures/15_17032026_UniversalCompressionIII_MinimaxRedundancy_study.md` says mixture distributions are optimal universal coding distributions under a prior; `lectures/5_04022026_Inequalities_study.md` says every bit of cross-entropy excess is mismatch cost.

## Course-material connection
The interpolated predictor is a small model average over Markov orders. `C` controls how quickly the model trusts a more specific context. Raising `C` is a more conservative universal-coding mixture: it protects against context-order mismatch by retaining more lower-order probability mass unless high-order counts are very strong.

## Hypothesis
If Run 009 worked because hard backoff overtrusted noisy order-3 contexts, then `C=32` may improve calibration further by mixing lower-order evidence for longer. If `C=16` was already near optimal, `C=32` may blur useful high-order signal and worsen.

## Risks
Runtime remains roughly the same as Run 009 but about twice the baseline. The public-practice improvement from Run 009 had low confidence according to autoresearch, so tuning constants on the same set risks overfitting; any new best should later be checked on train-derived validation.

Run:

```bash
PREDICTOR_PATH=submissions/ngram_interpolated_n4_c32.py ./autoresearch.sh
```

Success means `bits_per_symbol < 2.9817296585` for a new best, with `timed_out=0`, `evaluated_tokens=200000`, and automatic checks passing.
