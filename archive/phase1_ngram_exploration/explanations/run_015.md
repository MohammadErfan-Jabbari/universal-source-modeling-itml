# Run 015 — interpolated n=4 with C=44

## Proposed change
Add `submissions/ngram_interpolated_n4_c44.py`, a local interpolation-constant variant using `C=44`. It lies between current-best Run 013 (`C=40`) and Run 012 (`C=48`).

## Source and evidence
- source: experiment observation — Run 013 (`C=40`) is current best at `2.9792409029`; Run 014 (`C=36`) worsened to `2.9793094026`; Run 012 (`C=48`) scored `2.9792935749`.
- source: experiment observation — The local optimum appears close to C=40, but public-practice differences are tiny and need bracketing/confirmation.
- source: ideas backlog — `autoresearch.ideas.md` proposes count-dependent context-order mixtures `count/(count+c)`.
- source: course material — `lectures/15_17032026_UniversalCompressionIII_MinimaxRedundancy_study.md` motivates mixture distributions for robust universal coding; `lectures/5_04022026_Inequalities_study.md` interprets cross-entropy excess as mismatch cost.

## Course-material connection
The interpolated predictor mixes Markov-order models instead of selecting one hard context. `C=44` tests a slightly more conservative mixture than C=40, retaining lower-order probability mass for longer. This probes whether the robust mixture should be just above or below the current best.

## Hypothesis
If the true/local optimum lies between C=40 and C=48, `C=44` may improve. If C=40 is already optimal or the C=40 advantage is just noise, C=44 will match or worsen slightly.

## Risks
The likely metric difference is much smaller than the noise floor, so a small change should not be over-interpreted. Continued public-practice tuning risks overfitting; later train-derived validation is mandatory. Runtime and validity should be unchanged.
