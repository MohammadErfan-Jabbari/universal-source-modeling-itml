# Run 024 — Selective online updates for count-dependent CTW

## Proposed change
Add `submissions/selective_ctw_vomm_ppma_hybrid.py`. It keeps the PPM-C/Phase1 base unchanged, keeps the confirmed selective VOMM and selective PPM-A policies from Run 022, and changes only the count-dependent CTW component: train fitting populates all depths, but public-practice online updates affect only depth 0 and the deepest available context.

## Source and evidence
Runs 019 and 022 showed selective online updates help PPM-A and probabilistic-depth VOMM. Run 021 showed they slightly hurt PPM-C, so the lesson is component-specific. CTW is the remaining tree expert in the stack. Because CTW recursively mixes shorter and longer contexts, intermediate-depth online updates may either stabilize the tree or inject noisy mismatch.

Course evidence: Lecture 4's cross-entropy/KL identity says a wrong conditional distribution costs extra bits; update policy matters only through how it changes that mismatch. Lecture 14's minimax redundancy section frames universal coding as robustness against source uncertainty; selective CTW updates are a robustness choice about which parts of the context tree should adapt online.

## Course-material connection
The predictor remains strictly sequential: all public symbols are incorporated only after scoring. The experiment asks how a context-tree coding distribution should synchronize with the observed stream: all tree nodes along the suffix path, or only the global and most-specific memories.

## Hypothesis
If intermediate-depth CTW public updates are noisy, selective CTW should improve the confirmed Run 022 stack. If CTW's small gain depends on smooth all-depth online adaptation, this will regress. Since CTW was an early contributor to Phase 2 improvements, this is the last natural selective-update test before runtime consolidation.

## Risks
Selective updates already failed for PPM-C, so CTW may also need all-depth updates. Current best margins are extremely tiny, so any improvement needs confirmation. The model is still slow because it preserves the full stacked expert architecture.
