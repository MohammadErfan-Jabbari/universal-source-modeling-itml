# Run 057 — Stronger support-conditioned disagreement boost

## Proposed change
Add `submissions/strong_support_conditioned_disagreement_boost_ppma_stack.py`. It keeps the confirmed Run 054 architecture and increases the support-conditioned disagreement boost:

```text
Run 054: multiplier = 1 + 0.52 * JS * sqrt(count_conf), cap 1.44
Run 057: multiplier = 1 + 0.60 * JS * sqrt(count_conf), cap 1.52
```

All experts, count tables, escape laws, and online update policies remain unchanged.

## Source and evidence
Run 054 improved public practice beyond Run 052, and Run 055 confirmed it. Run 056 showed a pure scalar high boost improves train-val but misses the support-conditioned public score, suggesting support conditioning changes public calibration rather than merely increasing the average boost. This run tests whether the support-conditioned family can still safely increase the boost.

Course evidence: Lecture 4's conditional mutual information and chain-rule sections say an added expert is valuable by the uncertainty reduction it contributes after the baseline is known. JS disagreement proxies non-redundant PPM-A information, while `sqrt(count_conf)` proxies reliability of that recurrence evidence. Lecture 13's recurrence view supports using stronger repeated-context evidence more than weak recurrence.

## Course-material connection
The predictor remains strictly sequential: train counts and past public symbols only, with public updates after scoring. This is deterministic mixture-gate arithmetic, not a data-access or evaluator change.

## Hypothesis
If support-conditioned disagreement still underweights reliable PPM-A incremental signal, increasing boost to `0.60` should improve over Run 054. If Run 054 is near the calibration sweet spot, this will over-amplify supported but mismatched contexts and regress.

## Risks
Run 054 already showed a public/train-val tradeoff, so overfitting risk is real. Keep only on official primary improvement and confirm immediately.
